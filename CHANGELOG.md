### Added

- **Real summary renderer (lecture- and topic-scoped):** [`summary_render.py`](backend/app/services/answers/summary_render.py) now drives the summary mode independently of chat. **Lecture-scoped** path keeps a hard `lecture_number` filter, orders chunks by `chunk_order` / `position` / `order` metadata when present (forward-compatible — `LectureChunk` doesn't carry this column today) and falls back to `id` ascending; dedupes section heading prefixes via `_topic_head` so repeated MFCC subsections collapse to one bullet; renders `Summary: Lecture N` + `### Main idea` / `### Key topics` / `### How the topics connect` / `### Study focus`. **Topic-scoped** path filters chunks down to those whose `topic` / `keywords` / `clean_explanation` mention the canonical topic *or* one of its `ConceptKB` aliases (so a "recap of MFCCs" never surfaces formant or softmax chunks even when retrieval pulled them in), then renders `Summary: <topic>` + `### Core idea` / `### Key points` / `### Study focus` (`Core idea` instead of `Main idea` per the spec). Both paths drop `Course Answer` headings entirely. Tests: [`test_answers_summary_render.py`](backend/tests/test_answers_summary_render.py) covers `Summarize Lecture 10`, `Main takeaways from Lecture 10`, `What are the main ideas of Lecture 16?` (drops other-lecture chunks), `Give me a recap of MFCCs` (drops unrelated softmax chunks), topic dedupe, `chunk_order` metadata ordering, lecture-only fallback when retrieval is empty, and the topic layout's `### Core idea` (replacing the old `### Main idea`).
- **Compare evidence bundles V2 (entity-separated):** New [`ConceptEvidenceBundleV2`](backend/app/services/answers/entity_retrieval.py) dataclass carries the full V2 surface — `concept`, `aliases`, `evidence_chunks` (full chunk dicts, not just ids), `core_lines` (entity-pure after forbidden + cross-entity filtering), `support_score`, `forbidden_hits`, `shared_lines`, `source_metadata` (lectures + topic heads seen), `confidence`. Legacy callers keep working: V2 exposes `concept_id` / `label` / `chunk_ids` / `gap_flags` as read-only properties, and `from_legacy_bundle()` / `to_legacy_bundle()` adapters round-trip through the original four-field bundle. New `EvidenceBundleLike = ConceptEvidenceBundle | ConceptEvidenceBundleV2` type alias is what `AnswerPlan.evidence_bundles` now stores. New builder `build_bundles_for_compare_v2` (and multi variant `build_bundles_multi_v2`) per-line classifies each unit against both entities' term sets via `classify_line_for_compare`: a line goes to side A's `core_lines` only when it scores strictly higher for A than B and isn't blocked by an A-side forbidden term; lines that score >= 1 for both entities at a min-ratio threshold land in `shared_lines` on **both** bundles. [`compare_render.format_two_entity_compare_markdown`](backend/app/services/answers/compare_render.py) now reads `core_lines` directly off V2 bundles (skipping the `scoped_lines_from_chunks` recompute) and emits a new `### What they share` section when both bundles' `support_score` clears `COMPARE_SHARED_MIN_SUPPORT` (0.25) and shared lines exist; the section is omitted entirely otherwise (no placeholder text). `format_multi_entity_compare_markdown` shares the V2 `core_lines` source switch but does not emit a shared section (3+ entity semantics get fuzzy). [`answer_planning.build_answer_plan`](backend/app/services/answers/answer_planning.py) is now wired to the V2 builders for both two-way and multi-entity compare. Tests: [`test_compare_render.py`](backend/tests/test_compare_render.py) — `Compare CNN and MLP` (no convolution leak into MLP, no fully-connected leak into CNN; line-level disjoint), `CNN vs transformer` (no self-attention in CNN core_lines, no convolution in transformer core_lines), `Difference between MFCCs and formants` (cepstrum/filterbank stay on MFCC, vocal tract stays on formant), `Bias versus variance` (shared sentence surfaces under `### What they share`), `Contrast softmax and hardmax` (probability-distribution language stays on softmax, argmax/one-hot stays on hardmax, `core_lines` disjoint), shared-section-omitted when bundles are disjoint, and `from_legacy_bundle` ↔ `to_legacy_bundle` round-trip preserves all four legacy fields plus pulls aliases off the KB.
- **Tutor-style chat-mode renderer:** New `render_tutor_style_answer(plan, evidence)` in [`answer_generation.py`](backend/app/services/answers/answer_generation.py) replaces the legacy `### Direct Answer / Explanation / Example / Why it matters` markdown for chat-mode intents (`direct_definition`, `multi_step_explanation`, `scoped_explanation`, `simplified_reteach`, `teaching_plus_check`) with a flowing tutor narrative. The only explicit label kept in the output is **`The key idea:`** — opening sentence and contrast / clarification flow as short paragraphs (1–2 sentences each, blank line between), the example block is visually separated and lifts numeric arrays (e.g. `[0.12, 0.88]`) onto their own line, and the closer is a short grounded `That matters because…` sentence tied to the concept and topic (no `"You'll keep running into …"` / `"This topic connects to …"` filler). Content cleanup runs before rendering: generic filler is stripped, sentences already used in the opening cannot reappear in contrast / key-idea, and the final answer is paragraph-deduped. **Fallbacks never force structure:** when no contrast cue appears in the explanation lines the contrast block is skipped, and when only the legacy `"Think of the explanation above as the core picture…"` placeholder is available (no `sample_answer`, paired sample question, or usable source excerpt) the example block is skipped via `_has_concrete_example`. `no_examples` and `intuition_only` keep the tutor narrative but suppress the example block (via call-site `dataclasses.replace`, planner untouched); `exact_explanation_count`, `repeat_explanation_times`, and the `allow_incorrect_statements` safety refusal keep the legacy structured layout because they explicitly request the numbered / repeated / safety copy. Compare, compare_multi, lecture_summary, cross_lecture_synthesis renderers are unchanged. No retrieval, planning, or validation logic changed; no new LLM dependencies. Tests: [`test_structured_pipeline.py`](backend/tests/test_structured_pipeline.py) `test_chat_mode_uses_tutor_narrative_format`, `test_chat_mode_drops_legacy_section_labels`, `test_chat_mode_natural_paragraph_flow`, `test_chat_mode_softmax_example_and_no_repeated_lines`, `test_chat_mode_skips_example_block_when_no_concrete_example`, `test_chat_mode_no_examples_uses_tutor_narrative_without_example_block`, `test_chat_mode_repeat_explanation_keeps_legacy_layout`.
- **Per-mode renderer dispatch (chat / quiz / compare / summary):** `effective_mode` now drives the actual output shape, not just retrieval. New rule-based renderers [`quiz_render.py`](backend/app/services/answers/quiz_render.py) (emits `Quiz: …`, up to 3 questions in short / multiple-choice / true-false formats with an inline `Answer Key:`, never Course Answer headings) and [`summary_render.py`](backend/app/services/answers/summary_render.py) (lecture-scoped layout `Summary: Lecture N` + **Main idea** / **Key topics** / **How the topics connect** / **Study focus**, plus a compact topic-scoped variant) wired into [`answer_generation.py`](backend/app/services/answers/answer_generation.py) parallel to the existing compare branch. [`course_generation.py`](backend/app/services/generation/course_generation.py) forces rule-based generation for `compare`, `compare_multi`, `lecture_summary`, and `teaching_plus_check` so no LLM path can overwrite the deterministic per-mode shape. [`retrieval_v2.py`](backend/app/services/retrieval_v2.py) gains `_handle_quiz` so lecture-scoped quiz queries (e.g. "Test me on Lecture 11") pull lecture-filtered chunks via `_handle_summary` while topic queries keep definition-style retrieval. ### Fixed

- **Reset-password 500 on valid tokens:** [`reset_password`](backend/app/routes/auth.py) compared `PasswordResetToken.expires_at` (naive `db.DateTime`, returned naive by SQLite) against a timezone-aware `datetime.now(timezone.utc)`, raising `TypeError: can't compare offset-naive and offset-aware datetimes`. Naive read-backs are now normalized as UTC before comparison. Email-verification path (`consume-verify`) already compared as naive and is unaffected.
- **Alembic:** migration **`003`** `down_revision` corrected to **`002_chunk_key`** (was invalid **`002`**). Migration **`001`** `response_variants.retrieval_log_id`: named **`create_foreign_key`** for SQLite batch mode (fixes **“Constraint must have a name”** on `flask db upgrade`).

### Known limitations

- **Gemini boost** uses the Google Generative Language REST API; model availability and IDs vary by account — set **`GEMINI_MODEL`** accordingly. If both Gemini and OpenAI calls fail, no Boosted Explanation is returned even when gating says boost.
- **Structured pipeline** validation is heuristic (lexical / pattern checks); **study** routes still use the legacy retrieval path until opted in.
- **Embedding / hybrid** retrieval requires **`EMBEDDING_RETRIEVAL_ENABLED`**, **`OPENAI_API_KEY`**, and **`flask --app wsgi embed-chunks`**; large corpora may need batching / Postgres for scale.
- **Synthesis** diagnostics refer to the primary lexical pass; after diversification and deduplication, displayed chunk order may not match diagnostic rank order one-to-one.
- **Compare** side queries use the extracted concept strings (plus optional lecture hint) to limit cross-talk; unusually short extractions can yield thinner lexical matches than the full question would.
